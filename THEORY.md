# SecondBrain Catalog · 理论依据

> 货架式知识存储方案的理论基础。本文是去个人化提炼，方法论中立、可移植。
> 工程产物（规则 skill `SKILL.md`、脚手架 `framework/`）均由本理论推导而来。

---

## 〇、心智模型：逻辑物流仓储 + 地图 ≠ 政府

把第二大脑想成一套**逻辑物流仓储系统**。核心分界线：**包裹外部（地址 + 标签）结构化；包裹内部不定义。**

| 物流概念 | 我们的层 | 编号/命名 | 是否治理 |
|---------|---------|----------|:---:|
| 城市 | L1 顶级领域 | 十位区 `10/20/30` | ✅ |
| 厂区/园区 | L2 次级领域 | 局部 `01/02/0A`，语义按域声明 | ✅ |
| 楼栋 | L3 主题/工作单元 | 局部 `01/02` 或 项目 ID | ✅ |
| 货架/货位 | L4 子主题（可选） | 局部 `01/02` | ✅ 治理到此为止 |
| 包裹外标签 | 路径地址 + 文件名 | 命名规范承重 | ✅ |
| **包裹内部** | 工作单元/仓库内部 | 功能层 overlay + 自由文件 | 🚫 **故意不定义（自治）** |
| 仓库（物理） | 各设备 | 各持 catalog 子集 | ✅（跨设备同步） |

两条解耦原则：

- **地图 ≠ 现场**：catalog = 家 / 单一事实来源（SSOT）；实际编辑检出发生在各运行时工作区。App 持全量数据，第二大脑只存沉淀。
- **地图 ≠ 政府**：catalog = 疆域（城市→货架，谁在哪，稳定）；Agent / 运行时 = 在疆域上运行的班子（可重构）。**设计驱动运行时重构，而非反过来。**

## 一、设计原则 P1–P10

| # | 原则 | 含义 |
|---|------|------|
| P1 | 目录+文件名承重 | 分类靠目录与文件名（binary 同样有效）；frontmatter 仅 md 增强 |
| P2 | 工具/运行时中立 + 纯文本/git 友好 | 全文件存储；人与多种 Agent 直接读写 |
| P3 | 多轴正交，folder 是投影 | 物理目录承载主轴，其余靠编号/文件名/frontmatter |
| P4 | 可扩展 = 加规则不重构 | 加领域=加目录；加场景=契约加一行 |
| P5 | 名称自解释 + 按需创建 | 编号+描述名；用到才建，不铺空壳 |
| P6 | 来源/用途分层，不按"谁写"分层 | "谁写/私不私"靠元数据 |
| P7 | 唯一归属 | 同一种东西只能一个合法位置 |
| P8 | 层级即价值（重要性优先、热度其次） | 深度与价值成反比 |
| P9 | 地位向下传导 | 父节点层级决定子树地位下限 |
| P10 | 纵向定价、横向分类 | 深度=价值定级；同层内=按类别排布 |

## 二、结构规则 R1–R5（树的约束）

| 编号 | 规则 | 判定 |
|------|------|------|
| **R1** | 治理层（L1–L4）每段目录名必须是 `NN-name` | 合法名 = `^[0-9A-Z]{2}-.+`（两位数字/base36 大写 + 连字符 + 描述名） |
| **R2** | 项目 ID / 工作单元（包裹）**只能**在最深货架之下，禁止当治理层名 | 工作单元 = 包裹，不是货架 |
| **R3** | L1/L2/L3 强制存在；L4 可选 | 任何内容至少落在三层治理之下 |
| **R4** | 物理仓库根不计入逻辑编号 | 根目录不需要 `NN-` |
| **R5** | 年份固定 4 位；编号两位（base36 大写）；同一排序轴只用一种记法；存量货架（编号）与流量货架（年份）可并存一层 | 禁 `26` / `2026-06` / 小写 `0a` |

**例外**：年份货架（`2025/2026`）是 R5 允许的自排序值，免 `NN-`；以下划线开头的内部辅助目录（`_guide.md`/`_config`）是元文件/包裹，不是治理层。

## 三、操作规则 O1–O6（动手纪律）

> R 规则回答「树该长什么样」，但不回答「动手时如何保证不违规」。
> 实践证明：只有 R 规则时，人会先干活、后校验，于是反复违规。
> O 规则把「校验」前置成「动手的前置闸门」。**在治理工作里，校验就是工作。**

| 编号 | 规则 | 含义 |
|------|------|------|
| **O1** | 建前校验闸 | 任何创建治理层的 `mkdir`/`mv` 前，逐段校验每个新建路径段满足 R1，否则不执行 |
| **O2** | 包裹不可侵犯 | 项目/工作单元/用户资产 = 包裹，无权改名/移动/删除，只能在其外围建货架 |
| **O3** | 规格先行 | 先改规格、再动目录；目录是规格的投影，不允许目录跑在规格前 |
| **O4** | 不建空壳 | 只按规格深思熟虑建货架；禁止占位/猜测/重复目录 |
| **O5** | 动手前出清单 | 批量 `mkdir`/`mv` 前展示完整目标路径清单，让前缀肉眼可查再执行 |
| **O6** | PII 铁律 | 家庭/健康/财务等隐私域只存本机，禁提交任何 public git |

**自检清单**（每次动手前逐条过，任一不过 → 停）：

- [ ] O3 规格里有这个结构吗？还是临时发明的？
- [ ] O2 要动的是货架（治理层）还是包裹（用户资产）？包裹一律不碰。
- [ ] O1 每段新目录名都是 `NN-name` 吗？
- [ ] R3 建完后内容落在 ≥3 层治理之下吗？
- [ ] O5 完整路径清单给用户看了吗？
- [ ] O6 这是 PII 域吗？有无被推上 git 的风险？

## 四、七条通用规则

1. **一等公民由分类定**，不由热度定
2. **热度只改排序和 Dashboard 入口**，不改结构
3. **`00-` = 共同/通用席位**；禁 `misc/其他` 垃圾抽屉
4. **唯一归属**：归位(move)=放错移过来；演进(evolve)=孵化新性质 → 原件留存 + 新域新建 + 双向链接
5. **重要但稀疏 → 保留席位**，不降级、不铺空壳
6. **工具/现场 ≠ 地图**：App 持全量，第二大脑只存沉淀
7. **时间货架看"什么在变"**：纯流水→年份；角色在变→历程（时间锚+角色名）

## 五、实体模型（四实体）

| 实体 | 角色 | 落地 |
|------|------|------|
| **项目 Project** | 单一事实来源(SSOT) | 一个独立 git 仓；远端=权威源 |
| **组织 Organization** | 所有权/归属（某时点唯一，可转移） | 路径表达；转移=移目录+记日志 |
| **Agent（人格角色）** | 行政主理领域 + 项目参与者 | 人格工作区；工具(claude/codex/…)是可换后端 |
| **工作区 Workspace** | 某 Agent/运行时的私有工作区（非权威） | 检出 + 草稿；多运行时并行 |

```
权威层 远端 git  ←→  分类层 catalog 疆域  ←→  工作区层 各 Agent/设备/运行时私有检出
```

聚合轴：存量（持有·靠"是什么"找）→ 类别；流量（流过·靠"哪一年"找）→ 年份。
五种轴：① 时间 ② 类别 ③ 关系 ④ 职能 ⑤ 归属。

内部规格为**权威源**（持续演进）；本文若与最新规格不一致，以规格为准并回灌同步。
