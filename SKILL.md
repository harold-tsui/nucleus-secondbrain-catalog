# SecondBrain Catalog · 组织规范与放置手册

> 这是一套**工具中立的目录规范**：定义一组"空货架"和"往上放东西的方法"。
> 任何 Agent（如 CDO）在任意电脑/服务器上整理第二大脑时，按本手册决策即可，无需重新设计。

---

## 0. 一句话心智模型

第二大脑 = **逻辑物流仓储**。`仓库(物理机) → 城市(L1领域) → 厂区(L2) → 货架(L3) → [货位(L4)] → 包裹(项目/工作单元)`。
**治理层（城市/厂区/货架）= 编号货架，稳定；包裹内部 = 自治，不管。**

- **地图 ≠ 现场**：catalog 是文件的"家/归属"；正在编辑的 git 检出在 `~/Workspaces/{runtime}`。
- **地图 ≠ 政府**：catalog 是疆域；Agent/运行时是在上面干活的班子，可换。

## 0.5 开工第一问：先确认仓库拓扑（建任何目录之前必做）

第二大脑 = **一套统一逻辑 catalog + 每台机器物化其子集**。设备不是目录层级，只是"检出了哪个子集"。所以在新机器上**动手前，先问客户**：

1. **有没有「中央库」？** = 完整副本/权威全集（数据中心、云盘全量如 `OPENCLAW_CLOUD_WORKSPACE`、master SecondBrain）。在哪？路径/地址？
2. **有没有「兄弟库」？** = 其他设备各持子集的**同一个逻辑 catalog**。有哪些？各持哪些域/子集？
3. **本机该持哪个子集？** 按角色定（笔记本=热项目；服务器=已上线；公司机=公司域）。
4. **怎么同步？**（git / 加密盘 / rsync / Syncthing…，见 TD-4；注意 PII 域走加密同步、不上公共 git）。

**据答案决定动作：**

| 情况 | 动作 |
|------|------|
| **有中央库** | **不从零建**。从中央库**克隆/同步**本机所需子集，**沿用其既有结构与编号**，不另起炉灶。 |
| **有兄弟库** | 登记进 `00-system/03-ops` 的设备/库注册表；协调子集边界，保证同一东西**跨库仍唯一归属**（不重复）。 |
| **都没有** | 本机即首个库（可担任中央库）。按 §2 空货架从零建，并在 `ops` 标注"我是 origin / 全集所在"。 |

> **为什么必须先问**：不问就建 = 制造孤岛/分叉/重复归属，直接违背唯一归属（P7）。宁可多问一句，不可建错根。

## 1. 结构铁律 R1–R5

- **R1** 治理层（L1–L4）一律 `NN-描述名`；**禁止裸项目 ID 当层名**。
- **R2** 项目/工作单元只作**包裹**，放在最深货架之下，内部不中央治理。
- **R3** **L1/L2/L3 强制存在**；L4 可选（内容多才铺）。
- **R4** 仓库根（如 `~/SecondBrain`）是物理机器，跨机各持子集，不计入逻辑编号。
- **R5** 命名：`NN-` 两位 base36 大写（溢出用 `0A/7Z`）；年份固定 4 位 `2026`；同一排序轴只用一种记法；存量货架(编号)与流量货架(年份)可并存一层（年份自然排后）。中英文：英文为主，家庭/社交/健康/兴趣域内可中文。

## 2. 空货架（12 域骨架 · 直接照建）

```
00-system/        系统区:inbox(唯一收件口)/dashboard(MOC)/governance(规范模板)/ops(脚本)
—— 8 生命领域（重要性领先排序，每域可放 _guide.md）——
10-family/        家庭:00-共同体 + 各成员(按亲属关系平铺)   L3=事项类型
20-social/        人际:00-人脉经营 + 关系档案(按来源圈层) + 社群   轻域,只存沉淀
30-health/        健康:00-共同(家族/医疗资源) + 各成员(person)   L3=健康事项
40-self/          自我:overview/values/goals/journal
50-learning/      学习:00-体系 + 在攻(目标专题) + 修行(学科)
60-finance/       财务:总览/资产/负债/现金流/投资/保险/规划/税务
70-career/        事业:职业自我 + 我的事业(owned) + 任职(employed) + 共享业务网络
80-lifestyle/     兴趣:00-生活方式 + 各爱好(运动/旅行/娱乐/饮食…)
—— 功能层 ——
A0-agents/        人格工作区(每persona一槽,内含AGENTS/MEMORY/tasks/scratch/checkouts.json)
B0-knowledge/     跨域知识(被≥2域引用才上;主题分类)
C0-archive/       冷档案(保留原归属路径)
```

**按需创建，不铺空壳**：货架用到才建。`00-` 永远留给该层的"共同体/通用/元"。
**PII 铁律**：`10-family / 30-health / 60-finance` 含隐私 → **只存本机，禁止提交 git**。

## 3. 放置决策流程（拿到一个文件/项目，依次问）

> **首次在一台机器上动手前，先做 §0.5 开工第一问（确认中央库/兄弟库）。** 拓扑确认后，再按下面放具体文件。

1. **是隐私吗？**（家庭/健康/财务/证件）→ 进对应本机域，禁 git。
2. **它当下主要被当作"什么"来检索？** → 定 **L1 域**（这是唯一归属的总判据）。
3. **域内它属于哪个稳定主体？**（某人/某组织/某对象/某职能）→ 定 **L2 席位**。
4. **同类东西怎么聚合找回？** → 定 **L3 货架轴**（见 §4）。
5. **它是项目/工作单元吗？** → 作**包裹**放最深货架下；包裹内部用功能层 `1_raw/2_work/3_wiki/4_output` 自治。
6. **跨域/边界模糊？** → 唯一归属（§5），一处放、他处留引用。

> 找不到合法位置时，进 `00-system/00-inbox` 暂存，事后分诊——**绝不新建 `misc/其他` 垃圾抽屉**。

## 4. 怎么选 L3 货架轴

**先判存量 / 流量：**
- **存量**（你*持有*的、靠"是什么"找）→ 按**类别**。例：资产、保单、档案、收藏。
- **流量**（*流过*你的、靠"哪一年"找）→ 按**年份**。例：账单、日记、体检、项目年份。

**5 种可选聚合轴**（按"内容实际被怎么找回"选）：
① 时间(年份/历程) ② 类别(类型/学科) ③ 关系(圈层/对象/来源) ④ 职能(固定槽位) ⑤ 归属(原路径,归档用)。

**时间轴的切法**看"什么在随时间变"：纯流水/身份不变→年份；随时间变的是角色→历程（时间锚+角色名，旧阶段不可变）。

## 5. 七条通用规则（决策启发式）

1. **一等公民由分类定，不由热度定**：L2/L3 席位 = 稳定主体；热门不新增席位。
2. **热度只改两处**：同层 `01-` 排序 + Dashboard 入口（不改结构）。
3. **`00-` = 通用/兜底席位**：收"不属于任一主体"的；新类型加货架，不重构。
4. **唯一归属 + 让渡**：一物一家。**归位(move)**=放错了就移+记日志；**演进(evolve)**=孵化出新性质的东西→原件留存、新域新建、双向链接，**不物理移动**（防断链）。
5. **重要但稀疏 → 保留席位 + 按需创建**：层级由重要性定，不由文件量定；重要的域哪怕空着也占位，不降级、不铺空壳。
6. **工具/现场 ≠ 地图**：外部 App（通讯录/相册/游戏库）持全量；第二大脑只存其装不下的**沉淀**，不镜像搬运。
7. **形式 ≠ 分类**：网页/PDF/视频等载体不单独建文件夹，进包裹的 `1_raw/` 按主题分；大文件走冷热分层（只存索引卡+外链）。

## 6. 如何判断一个文件夹的"地位"

地位 = **价值定级**，由位置决定，供 Agent 判断该投入多少、是否该浅出/归档：

- **越浅越重要（P8）**：顶层 = 高价值高频；越深 = 越低频/越接近档案。深度 = 访问代价 = 价值。
- **领域重要性有序**：关系/健康/自我（根基）领先；**事业是手段不是目的**，重要但不挤占根基的位置。
- **席位 vs 包裹**：L2/L3 是"一等公民席位"（稳定、有地位）；包裹（具体项目）地位随其所在货架与生命周期浮动。
- **地位向下传导（P9）**：父节点地位决定整棵子树的地位下限；提升父节点 = 提升子树。
- **热度 ≠ 地位**：再忙的项目也不因热度获得更高结构地位，只在 Dashboard 拿快捷入口。
- **判断某项目文件夹地位时**：看①它在哪个域（域的重要性）②它在第几层（越浅越重）③它是活跃/沉淀/封存哪个温度。

## 7. 各类项目（包裹）落哪

无统一"projects"文件夹。**项目落在它服务的域之下作包裹**；要总览所有在跑项目，靠 `00-system/dashboard` 的 MOC 引用聚合。

| 项目性质 | 落点 |
|---------|------|
| 自有事业项目 | `70-career/01-我的事业/{事业体}/{年}/包裹` |
| 雇主岗位项目 | `70-career/02-任职/{雇主}/{岗位历程}/包裹` |
| 研究/学习专题 | `50-learning/01-在攻/{专题}/`（功能层包裹） |
| 健康/财务/家庭计划 | 对应生命领域内 |
| Agent/系统项目 | `A0-agents/{persona}/` |
| 完成归档 | `C0-archive/projects/{原归属}/{项目}-{年}` |

## 8. 放置前自检清单

- [ ] 这东西**唯一**的家是这里吗？别处有没有重复？
- [ ] 隐私域没被推上 git？
- [ ] 治理层都是 `NN-名`？项目 ID 只出现在包裹层？
- [ ] L1/L2/L3 三层齐了？没硬铺空 L4？
- [ ] 选的 L3 轴符合"它怎么被找回"？
- [ ] 跨域的，他处留引用/链接了吗？
- [ ] 没有制造 `misc/其他` 抽屉？
